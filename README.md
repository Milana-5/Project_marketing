# Проект "Предсказание покупки"

**1. Описание проекта**

В рамках проекта рассматриваются данные интернет-магазина. Магазин собирает историю покупателей, проводит рассылки предложений и
планирует будущие продажи. Для оптимизации процессов надо выделить пользователей,
которые готовы совершить покупку в ближайшее время.

**Цель проекта:**

* Разработать модель машинного обучения для предсказания вероятности покупки пользователем в течение 90 дней.

**2. Описание данных**

Исходные данные представлены в 5 файлах: **apparel-purchases.csv**,  **apparel-messages.csv**, **apparel\_target\_binary.csv**,  **full\_campaign\_daily\_event.csv** и **full\_campaign\_daily\_event\_channel.csv**.

Первый файл **apparel-purchases.csv** содержит данные о покупках клиентов по дням и по товарам. В каждой записи покупка определенного товара, его цена, количество штук. В таблице есть списки идентификаторов, к каким категориям относится товар. Часто это вложенные категории (например автотовары-аксессуары-освежители), но также может включать в начале списка маркер распродажи или маркер женщинам/мужчинам. Нумерация категорий сквозная для всех уровней, то есть 44 на второй позиции списка или на третьей – это одна и та же категория. Иногда дерево категорий обновляется, поэтому могут меняться вложенности, например \['4', '28', '44', '1594'] или \['4', '44', '1594'].

Параметры:

* client\_id - идентификатор клиента
* quantity - количество единиц товара
* price цена - товара
* category\_ids - идентификаторы категорий
* date - дата покупки
* message\_id - идентификатор сообщения из рассылки



Второй файл **apparel-messages.csv** содержит рассылки, которые были отправлены клиентам из таблицы покупок.

Параметры:

* bulk\_campaign\_id - идентификатор рассылки
* client\_id - идентификатор клиента
* message\_id - идентификатор сообщения
* event - действие с сообщением (отправлено, открыто, покупка…)
* channel - канал рассылки
* date - дата действия
* created\_at - дата-время полностью



Третий файл **apparel\_target\_binary.csv** содержит целевой признак.

Параметры:

* client\_id - идентификатор клиента
* target - клиент совершил покупку в целевом периоде



Четвертый файл **full\_campaign\_daily\_event.csv** содержит агрегацию общей базы рассылок по дням и типам событий.

Параметры:

* date - дата
* bulk\_campaign\_id - идентификатор рассылки
* count\_event\* - общее количество каждого события event
* nunique\_event\* - количество уникальных client\_id в каждом событии
* в именах колонок есть все типы событий event



Пятый файл **full\_campaign\_daily\_event\_channel.csv** содержит агрегацию по дням с учетом событий и каналов рассылки
.

Параметры:

* date - дата
* bulk\_campaign\_id - идентификатор рассылки
* count\_event\* - общее количество каждого события event по каналам
* nunique\_event\* - количество уникальных client\_id по событиям и по каналам
* в именах колонок есть все типы событий event и каналов рассылки channel



**Условия выполнения проекта:**

**4. Результаты исследовательского анализа.**



**Временной диапазон**

Исследовались данные покупок пользователями товаров интернет-магазина в период с 15 мая 2022 года по 16 февраля 2024 года.



**Параметр "target" (целевой признак)**

Число пользователей, которые совершат повторную покупку в течение 90 дней, составляет 2% от общего объема данных. Таким образом, в данных наблюдается дисбаланс классов.



**Параметр "count\_purchase"**

Параметр "count\_purchase" (количество покупок пользователя) имеет значения от 1 до 52. При этом 46% (почти половина) пользователей совершили за все время только одну покупку, 20% пользователей совершили 2 покупки и 14% пользователей - 3 покупки.

Всего 5% пользователей совершили 7 и более покупок за два года.

Видим, что даже пользователи из ТОП-5 пользователей по числу покупок совершали покупки неравномерно на протяжении всего двухлетнего периода.

**Параметры "mean\_purchase" и "total\_sum"**

Параметр "mean\_purchase" (средняя стоимость одной покупки) имеет значения от 1 условной единицы до 53 499. Медиана составляет 1 259, среднее значение 1 502. Средняя стоимость 5% всех покупок не превышает 201 условной единицы. Средняя стоимость 95% всех покупок не превышает 3 499.

Параметр "total\_sum" (общая стоимость всех покупок одного пользователя) имеет значения от 1 условной единицы до 297 954. Медиана составляет 2 099, среднее значение 3 301. Общая стоимость всех покупок 5% пользователей не превышает 399 условных единиц. СОбщая стоимость всех покупок 95% пользователей не превышает 9 796.

Динамика покупок показывает общую тенденцию снижения объемов. Есть месяцы с провалом в объемах покупок и есть месяцы с пиком покупок. То есть покупки неравномеры по времени. Количество рассылок сообщений выросло к концу рассматриваемого периода времени, а объемы покупок - наоборот, снизились.



**Параметры товаров и их категорий**



ТОП-3 лидеров - товары с id "431", "420" и "432" (суммарно составляют 13% от общего объема данных).

ТОП-3 категории товаров по количеству покупок - "4", "28", "5562". К ним относится порядка 42% данных.



**Недавние и долгие пользователи (параметры "last\_days" и "period")**

В среднем пользователи сделали последнюю покупку в течение последнего года. 5 178 (порядка 10%) пользователей сделали покупку в течение последнего квартала.

75% пользователей делали покупки только один день в рамках рассматриваемого двухлетнего периода. Всего 5% пользователей имеют более, чем 6-месячный период покупок (период времени от первой до последней покупки).



**Действия пользователей и воронка продаж**

Пользователи почти в два раза чаще открывают сообщения по мобильному телефону, чем по электронной почте, но покупают после открытия сообщения наоборот - по мобильному телефону практически в два раза реже, чем по электронной почте.

Частота покупок по электронной почте составляет порядка 1% от количества разосланных сообщений, частота покупок по мобильному телефону составляет порядка 0.4% от количества разосланных мобильных сообщений.



**Корреляция параметров**

Существует достаточно сильная корреляция между событиями "рассылка" по электронной почте, "открытие" письма и "клик" по нему (коэффициенты корреляции порядка 0.89-0.92). Такая же сильная корреляция между событиями "клик" и "покупка" по электронной почте (коэффициент корреляции порядка 0.9).

Что касается канала мобильной связи, то здесь корреляция между событиями "рассылка", "открытие" сообщения и "клик" по нему гораздо ниже (коэффициенты корреляции порядка 0.63-0.74). Корреляция между событиями "клик" и "покупка" по мобильному сообщению чуть ниже, чем по электронной почте (коэффициент корреляции порядка 0.84).

Есть достаточно сильная положительная корреляция между количеством дней со дня первой покупки и количеством дней со дня последней покупки - параметры "first\_days" и "last\_days" (коэффициент корреляции 0.91). Это и понятно, потому что почти половина покупателей покупали что-то только в течение одного дня в рамках рассматриваемого двухлетнего периода.

Взаимосвязь между параметрами визуально не сильно зависит от значений целевого признака.



**5. Модель для прогнозирования повторной покупки.**

В рамках данного проекта для поиска модели использовалось машинное обучение с учителем.

Тип нашей задачи для поиска модели - бинарная классификация на основе как числовых, так и категориальных параметров. В данных есть дисбаланс классов.

В качестве метрики для оценки качества модели как основная используется метрика "roc-auc". Дополнительно рассчитывались следующие метрики: "accuracy", "precision" и "recall".

Сначала данные были разделены на две выборки (тренировочную и тестовую) в соотношении 0.8:0.2. Разделение было выполнено с учетом дисбаланса классов.

Далее был проведен поиск наиболее качественной модели с точки зрения основной метрики. В процессе поиска использовались следующие методы:

* были протестированы следующие модели: KNeighborsClassifier, LogisticRegression и CatBoostClassifier.
* были рассмотрены разные комбинации входных параметров (с учетом их важности и корреляции между собой и с целевым признаком).
* для поиска оптимальной модели использовались GridSearchCV с разными гиперпараметрами моделей, также использовалась кросс-валидация.
* масштабирование количественных параметров проводилось с использованием RobustScaler и MinMaxScaler.
* предобработка данных и поиск оптимальной модели велся в рамках единого пайплайна.



В коде оставлены только те входные параметры и гиперпараметры, которые привели к наиболее оптимальному с точки зрения основной метрики результату.

Анализ важности параметров и их влияния на целевой признак был проведен с помощью "shap".

Также была построена матрица ошибок и рассчитаны значения метрик "accuracy", "precision" и "recall" для разных значений порога вероятности для определения к какому классу отнести целевой признак.
